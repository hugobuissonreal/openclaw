Les couches

Chaque couche existe pour une raison précise. Le but n’est pas de multiplier les dossiers, mais de séparer proprement ce qui est partagé, ce qui est hiérarchique, ce qui est analytique et ce qui est purement technique.

Partagé

Global

Le socle commun : mémoire durable, contexte métier, règles système, état opérationnel, standards partagés et protocoles.

Pilotage

CEO

La couche du chef d’orchestre : arbitrage, vision d’ensemble, délégation, learnings transversaux et roadmap structurelle.

Supervision

Manager

Le niveau intermédiaire, utile seulement quand plusieurs workers doivent être pilotés sans surcharge du CEO.

Exécution

Worker

La couche spécialisée : un agent, un format, un pipeline, des métriques, une boucle d’amélioration locale.

Workspace

La structure n’est pas là pour faire joli. Chaque bloc a une fonction : aligner, exécuter, synthétiser, documenter ou maintenir.

/workspace
  • memory/ — journal quotidien et historique récent
  • state/ — état opérationnel actif
  • context/ — vérité métier partagée
  • system/ — règles système globales
  • agents/shared/ — standards partagés entre agents
  • agents/ceo/<name>_ceo/ — couche privée du CEO
  • agents/managers/<name>_manager/ — couche manager optionnelle
  • agents/workers/<name>_worker/ — couche privée de chaque worker
  • analytics/ — synthèses lisibles alimentées automatiquement
  • ops/ — protocoles et docs opérationnelles
  • infra/ — SQL, scripts, migrations et maintenance technique
  • .learnings/ — erreurs, corrections et leçons d’exécution

Supabase

La base commune ne remplace pas les docs. Elle stocke ce qui doit être structuré, historisé et comparé dans le temps.

Tables cœur
  • agents — registre des agents
  • content_items — table centrale enrichie du contenu
  • metric_snapshots — évolution des métriques
  • content_analyses — analyses successives
  • human_feedback — feedback humain
  • agent_learnings — learnings consolidés
  • run_logs — historique des exécutions
  • asset_registry — assets liés au contenu
Tables optionnelles
  • content_variants — variantes créatives si le workflow le justifie
  • experiments — formalisme expérimental si nécessaire

Qui lit quoi

Tous les agents ne lisent pas tout. La hiérarchie documentaire doit suivre la hiérarchie agentique.

Lecture

CEO

Lit le global, les analytics, les ops, sa couche privée et, à la demande, l’infra.

Lecture

Manager

Lit le global, sa couche manager, les analytics utiles et les synthèses des workers qu’il supervise.

Lecture

Worker

Lit le global utile, les standards partagés, sa couche locale et les protocoles ciblés si nécessaire.

But

Principe

Le bon agent lit le bon niveau d’information, au bon moment, sans être noyé dans le reste.
